نمای کلی

کیت توسعه جامع یک فریم ورک همه منظوره مدل - ویو - کنترلر به جهت بالا بردن سرعت توسعه و ساماندهی کل سیستم می باشد
این کیت شامل اجزای با کیفیتی از جمله گزینه های زیر می باشد :

- موتور دیتابیس قدرتمند که قادر به پشتیبانی از سه دیتابیس معروف (MySQL, SQLServer, SQLite) ، سه جایگاه دیتابیس در طول برنامه و ORM با عملکرد بالا که قادر به پشتیبانی CRUD و چندین عملکرد دیگر می باشد.

- سرویس کد امنیتی

- ایمیل سرور

- سرویس آپلود امن

- سرویس ارزیابی ، فیلتر و زدودن رشته

- سرویس زبان ساختار یافته

- سیستم مسیر یابی با عملکرد بالا ، با پشتیبانی از میان راهی (فرانت کنترلر)

- ورودی های مدیریت شده (پارامتر های URL ، متد GET ، متد POST و غیره)

- نشست های رمز گذاری شده و ساختار یافته

- کامپوننت های عیب زدایی از جمله دیباگر و دیباگ بار

و ...

ساختار پوشه ها

APP :

اپ دایرکتوری پایه اپلیکیشن شما رو تشکیل میده ؛ تمامی کنترلر ها ، مدل ها ، میان راهی و API های شما در این پوشه نگه داری میشه .

CONFIG :

تنظیمات اپلیکیشن ، تنظیمات نشست ، تنظیمات سرویس مسیر یاب و متغیر های مسیر در این پوشه نگه داری میشه .

DATABASE :

بک آپ های دیتابیس ، فایل های Migrate دیتابیس ، Schema های دیتابیس و فایل دیتابیس درون برنامه ای SQLite در این پوشه نگه داری میشه .

PUBLIC :

پوشه پابلیک از سمت کلاینت کاملا قابل دسترسی خواهد بود .
استفاده رایج اون می تونه نگه داری منابع سمت فرانت مثل جاوا اسکریپت ، CSS و ... باشه .

RESOURCES :

پوشه ریسورس نگه دارنده منابع اپلیکیشن شما از جمله زبان های از پیش تعیین شده است

ROUTES :

نگه دارنده مسیر های ثبت شده در اپلیکیشن شما

STORAGE :

این پوشه حاوی کش های اپلیکیشن شما می باشد

VENDOR :

دایرکتوری Composer ؛ نگه دارنده پکیج ها و استاندارد PSR4 برای Autoloading

VIEWS :

این دایرکیتوری حاوی ویو های می باشد که توسط موتور قالب ، رندر خواهند شد .

پیکربندی

برای شروع کار با رد فریم ورک ابتدا باید یک سری تنظیمات رو اعمال کنید ؛ اولین مرحله تولید کردن کلید امنیتی جدید برای رمز نگاری اپلیکیشن شما است .

برای این کار نرم افزار رد انالیتیکس رو باز کنید و بر روی منوی Project و تب Generate New Secret Key کلیک کنید

کلید امنیتی جدید شما آماده خواهد بود .

تمامی تنظیمات نشست در Config -> Session.php در دسترس خواهند بود


تمامی متغیر های مسیر در Config -> Path.php در دسترس خواهند بود

*** هرگز متغیر های مسیر مربوط به هسته فریم ورک (کامپوننت های آماده) را شخصی سازی یا حذف نکنید .

متغیر های مسیری که باید توسط توسعه دهنده تعریف شوند :

تعریف کردن متغیر های مسیر ضروری - مثال
// ROOT_DIRECTORY
define('ROOT_DIRECTORY', '/');

// PUBLIC_PATH
define('PUBLIC_PATH', '/public/');

// MYSQL_PATH - اگر از یک دیتابیس مای اسکیو ال استفاده می کنید
define('MYSQL_PATH', 'D:\Wamp\bin\mysql\mysql5.7.24\bin\mysql');

// MYSQL_DUMP_PATH - اگر از یک دیتابیس مای اس کیو ال استفاده می کنید
define('MYSQL_DUMP_PATH', 'D:\Wamp\bin\mysql\mysql5.7.24\bin\mysqldump');

// SQL_CMD_PATH - اگر از یک دیتابیس اس کیو ال سرور استفاده می کنید
define('SQL_CMD_PATH', '"C:\Program Files\Microsoft SQL Server\Client SDK\ODBC\130\Tools\Binn\SQLCMD.exe"');

// SQLITE_CLI_PATH - اگر از یک دیتابیس اس کیو لایت استفاده می کنید
define('SQLITE_CLI_PATH', ROOT_PATH . "database" . DS . "SQLite" . DS . 'sqlite3.exe');

تنظیمات مسیر یاب در Config -> Router.php در دسترس می باشد .
نحوه ست کردن صفحه پیش فرض خطای 404 :

تنظیم کردن مسیر پیش فرض
Router::setDefaultRoute(function () { http_response_code('404'); \Red\View::render("Red.NotFoundError"); });

نحوه ست کردن صفحه پیش فرض خطای 403 (این خطا زمانی رخ می دهد که اچ تی تی پی متد یا سطح دسترسی کاربر درست نباشد) :

تنظیم کردن خطای دسترسی
Router::register('error/403', 'GET,POST,PUT,PATCH,DELETE', function (){ http_response_code('403'); \Red\View::render("Red.AccessError"); }, 'All');

برای تنطیم کردن خطای دسترسی در وب سرور ، فایل .htaccess در دایرکتوری ریشه ای خودتون رو باز کنید و نام دامنه رو جایگزین دامنه خودتون کنید .

تنظیم کردن خطای دسترسی در وب سرور
// نام دامنه خودتون رو جایگزین لوکال هاست کنید
ErrorDocument 403 http://localhost/error/403

متغیر های محیط توسعه

دو راه برای تنظیم کردن متغیر های محیط پیش روی شماست :

1 - فایل Environment.json در دایرکتوری ریشه ای

2 - فایل Config.php در دایرکتوری Config
در شرایطی که شما راه دوم رو انتخاب کنید باید گزینه استفاده کردن از PHP Config رو فعال کنید :

فعال کردن گزینه PHP Config
Red::setPhpConfig(TRUE);

PROJECT - Name :

نام پروژه شما که در رد آنالیتیکس ، دیباگ بار و در دیگر ابزار ها نمایش داده خواهد شد .

PROJECT - Version :

ورژن اپلیکیشن شما که در رد آنالیتیکس ، دیباگ بار و دیگر ابزار ها نمایش داده خواد شد .

PROJECT - State :

این فیلد حاوی مرحله فعلی پروژه شما هست که میتونه یکی از سه گزینه زیر باشه :

1 - Production
این مرحله به معنی "در حال توسعه" می باشد ؛ در این حالت تمامی کامپوننت های توسعه از جمله دیباگر و دیباگ بار فعال خواهند بود .

2 - Maintenance
این مرحله به معنی "در حال نگه داری" می باشد ؛ از این رو تمامی کامپوننت های عیب یابی غیر فعال خواهند بود .

3 - Break :
این مرحله به معنی "وقفه نگه داری" می باشد ؛ در این حالت سرور غیر فعال خواهد بود .

PROJECT - SupervisorIP :

کاربری که آی پی آدرسش برابر با SupervisorIP باشد به عنوان Supervisor شناخته شده و در طول "وقفه نگه داری" می تواند پروژه را رصد کند .

PROJECT - SDK :

SDK اشاره به کیت توسعه استاندارد دارد که کیت توسعه مورد استفاده برای این فریم ورک Red Framework Enterprise Application می باشد.

PROJECT - Language :

زبان فعلی اپلیکیشن شما ؛ زمانی که اپلیکشن چند زبانه نوشته شود تمامی خطا ها و متن ها به این متغیر بستگی دارد و به اصطلاح Soft Text هستند .

PROJECT - Programmer :

نام برنامه نویس یا تیم برنامه نویس پروژه شما .

PROJECT - Encryption :

سیستم رمزگذاری متد POST ؛ این مقدار می تواند off یا on باشد .

PROJECT - SecretKey :

این متغیر حاوی Salt Key اپلیکیشن شما برای رمز گذاری است ؛ Salt اصطلاحی است که در کریپتوگرافی به کار می رود و در اصل به اضافه کردن Salt Key به الگوریتم استاندارد عمل Salting می گویند .

PROJECT - TimeZone :

موقعیت زمانی اپلیکیشن شما (PHP Timezone)

PROJECT - Cache :

برای عمل کش کردن این گزینه را برابر on قرار بدهید ؛
با این کار فریم ورک و موتور قالب برای صرفه جویی در استفاده از منابع اعمال روتین خود را کش می کنند

DATABASES Slot :

رد فریم ورک سه جایگاه دیتابیس خالی برای توسعه دهنده فراهم می کند که در بخش "موتور پایگاه داده" کامل توضیح داده خواهد شد .

Mail Server :

رد فریم ورک بستری برای ارسال آسان ایمیل (SMTP Mail Server) فراهم میاورد که در بخش "ایمیل سرور" کامل توضیح داده خواهد شد .

DEBUG - DebugBar :

دیباگ بار یک کامپوننت زیبا و کارامد در رد فریم ورک ست که به جهت کمک به توسعه دهنده برای برسی روند اجرای برنامه طراحی شده است ؛ مقدار این متغیر می تواند on یا off باشد .

DEBUG - Errors :

رد فریم ورک کامپوننت عیب یاب حرفه ای به جهت کمک به توسعه دهنده در روند عیب یابی برنامه فراهم میاورد ؛ مقدار این متغیر می تواند on یا off باشد .

مسیر یابی

مسیر در رد فریم ورک دارای خواص زیر است :

  • خود مسیر (URL) + پارامتر های مسیر (اختیاری و غیر اختیاری)
  • متد مسیر (که در واقع همان HTTP متد است)
  • اکشن مسیر (Controller@Method)
  • میان راهی های مسیر (فرانت کنترلر)

برای ثبت مسیر جدید دو راه پیش روی شماست :

1 - ثبت مسیر به صورت دستی از Routes -> Routes.php
2 - ثبت مسیر به وسیله رد آنالیتیکس


در مثال رو به رو ما یک مسیر به صورت دستی و GET متد با دو پارامتر (که یکی غیر اختیاری و دیگری اختیاری است) به همراه میان راهی ارزیابی اعتبار و یک مسیر با متد GET و POST که اکشن آن ابجکت است ثبت خواهیم کرد :

ثبت مسیر
Router::register("panel/{param1}{?param2}", "GET", "PanelController@panel", ["Authentication:admin"]);


Router::register("welcome/{name}", "GET, POST", function($name) { echo 'Welcome :' . $name; });

ثبت مسیر به وسیله رد آنالیتیکس :

کنترلر

کنترلر در الگو مدل - ویو - کنترلر نقش اصلی را بر عهده دارد ؛ یک کنترلر تمامی تصمیم گیری های نهایی و ارتباط بین فرانت کنترلر (میان راهی / درخواست) ، مدل و ویو را مدیریت میکند .

نام یک کنترلر در رد فریم ورک از ساختار نوشتاری CamelCase (حرف اول هر کلمه حروف بزرگ می باشد) بهره می برد و پسوند Controller را دارا می باشد .
نام فضای کنترلر App\Controllers می باشد
کنترلر جدید شما باید از کلاس کنترلر پایه ارث بری کند :

کنترلر جدید
< ? php
/** Red Framework Controller
* Generated By Red Analytics
*
* Date: 26/08/2019
* Time: 01:14:23
* @author RedCoder
*/

namespace App\Controllers;

use Red\Base\Controller;

class TestController extends Controller
{

}

تولید کردن کنترلر جدید به وسیله رد آنالیتیکس :

برای تولید کردن مدل تیک Generate Model رو بزنید .

نکته : کلاس کنترلر به طور خودکار یک نمونه از کلاس مدل (با همین نام ، در شرایطی که وجود داشته باشد) گرفته و در متغیر حفاظت شده model قرار می دهد .
در شرایطی که شا بخواهید از این نمونه استفاده کنید و نمونه جدیدی نسازید ، باید کد خودتون رو برای ماشین (در اینجا منظور IDE هست) قابل فهم کنید ؛ روش استانداردی که پی اچ پی برای این کار ارائه داده ، PHP Doc هست که در ادامه توضیح داده خواهد شد .

مثال :

کنترلری که توان مسیر یابی ابجکت مدل را دارد
< ? php
/** Red Framework Controller
* Generated By Red Analytics
*
* Date: 26/08/2019
* Time: 01:14:23
* @author RedCoder
*/

namespace App\Controllers;

use Red\Base\Controller;

class TestController extends Controller
{
 /**
 * @var TestModel $this->model
 */
 protected $model;
}

مدل

مدل در الگوی مدل - ویو - کنترلر نقش لایه ی میانی بین کنترلر و داده ها رو بر عهده دارد .

نام یک مدل در رد فریم ورک از ساختار نوشتاری CamelCase (حرف اول هر کلمه حروف بزرگ می باشد) بهره می برد و پسوند Model را دارا می باشد .
نام فضای مدل App\Models می باشد

مدل جدید

*
* Date: 26/08/2019
* Time: 01:49:10
* @author RedCoder
*/

namespace App\Models;

}

تولید مدل جدید به وسیله رد آنالیتیکس :

ویو

ویو ها مشابه قالب هستند ؛ تمامی ویو ها در دایرکتوری ویو نگه داری و توسط موتور قالب رندر می شوند .

موتور قالب به شما این توانایی رو میده تا به راحتی با یک ساختار دستوری مرتب تر میان تگ های HTML از PHP استفاده کنید .

در حال حاضر رد فریم ورک از موتور قالب Twig که موتور قدرتمند و سریعی هست بهره می بره .

تمامی مستندات موتور قالب در وبسایت رسمی Twig قابل دسترس عموم قرار دارد .

دلیل اصلی من برای انتخاب Twig به عنوان موتور قالب این بود که حدود 90 درصد IDE ها و توسعه دهندگان در حال حاضر قابلیت درک ساختار دستوری Twig رو دارند .

احتمال توسعه موتور قالب شخصی رد فریم ورک در ورژن های آینده هنوز باقی هست ؛ ولی من این اطمینان رو به تمام توسعه دهندگان میدم که حتی در صورت توسعه موتور قالب شخصی ، سینتکس موتور قالب فعلی باقی بمونه تا ساختار رد فریم ورک یک پارچه نگه داشته بشه .


تولید ویو به وسیله رد آنالیتیکس :


رندر کردن ویو :

نکته : کاراکتر "." عملکرد مشابه جدا کننده دایرکتوری را خواهد داشت .

رندر کردن ویو
View::render("Partials.Footer");

میان راهی

میان راهی (Middleware) در واقع نام دیگر فرانت کنترلر در الگوی مدل - ویو - کنترلر می باشد .

در مفهوم ساده تر یک میان راهی کلاسی هست که قبل از پردازش اکشن اصلی در روند مسیر یابی پردازش شده .

نام فضای میان راهی App\Middlewares می باشد

برای ساخت یک میان راهی جدید ابتدا باید کلاس انتزاعی Middleware رو ارث بری کنید و سپس متد انتزاعی run رو پیاده سازی کنید .

متد run بی نهایت پارامتر دریافت می کند و مقدار بازگشتی اون داده ای از نوع Boolean است .

توجه داشته باشید که اگر متد run مقدار بازگشتی FALSE رو برگرداند پردازش کل اپلیکیشن مختل خواهد شد و اگر مقدار بازگشتی TRUE را برگرداند اپلیکیشن به پردازش ادامه خواهد داد .

میان راهی جدید
< ? php
/** Red Framework Middleware
* Generated By Red Analytics
*
* Date: 26/08/2019
* Time: 05:25:17
* @author RedCoder
*/

namespace App\Middlewares;

use Red\Base\Middleware;

class Test extends Middleware
{
 public function run(... $parameters)  {
 // TODO: Implement run() method.
 }
}

تولید کردن میان راهی به وسیله رد آنالیتیکس :


ارسال دو پارامتر 'Admin' و 'Supporter' به میان راهی ارزیابی اعتبار کاربر :

ارسال پارامتر به میان راهی
Router::register("panel/{param1}{?param2}", "GET", "PanelController@panel", ["Authentication:admin,supporter"]);

سرویس ای پی آی

سرویس ای پی آی در واقع کلاسی هست برای دسته بندی مسیر هایی که ارائه دهنده API خاصی هستند ؛ نام فضای این کلاس App\APIs می باشد .

در مثال رو به رو متد API از کلاس APIService رو در فرایند مسیر یابی صدا خواهیم زد :

سرویس ای پی آی
Router::register("api/", "GET", "APIService@API");

موتور پایگاه داده

رد فریم ورک از موتور پایگاه داده قدرتمندی برخوردار است که از سه پایگاه داده معروف (MySQL, SQLServer, SQLite) و ORM با عملکرد بالا پشتیبانی می کند ؛
این موتور پایگاه داده قابلیت استفاده از سه پایگاه داده به طور هم زمان در طول اجرای برنامه فراهم میاورد .


برای شروع کار با موتور پایگاه داده رد فریم ورک ابتدا باید متغیر های محیطی را مقدار دهی کنید :

متغیر های میحیطی - جایگاه دیتابیس 1
"DATABASE_1":
{
"Status": "off",
"Driver": "mysql",
"Host": "localhost",
"Port": "3306",
"Name": "red",
"User": "root",
"Password": "red",
"Charset": "utf8",
"Backup": "off",
"BackupTime": "23",
"Migration": "off"
}

STATUS :

برای استفاده از این جایگاه پایگاه داده مقدار این فیلد رو برابر on قرار بدهید

DRIVER :

از اونجایی که رد فریم ورک از سه پایگاه داده پشتیبانی میکنه پس درایور پایگاه داده میتونه یکی از سه مورد زیر باشه :

برای دیتابیس مای اس کیو ال : mysql
برای دیتابیس اس کیو ال سرور : sqlsrv
برای دیتابیس اس کیو لایت : sqlite

HOST :

هاست پایگاه داده

PORT :

پورت پایگاه داده

NAME :

نام پایگاه داده

USER :

نام کاربری پایگاه داده

PASSWORD :

رمز عبور پایگاه داده

CHARSET :

Charset پایگاه داده (پیش فرض آن UTF-8 است)

BACKUP :

اگر مقدار این فیلد on باشد ، رد فریم ورک به طور خودکار از دیتابیس شما در زمان BackupTime ، بک آپ خواهد گرفت .

توجه داشته باشید که عملیات پشتیبان گیری توسط خط فرمان انجام خواهد شد ؛ پس باید قبل از پشتیبان گیری 4 متغیر مسیر زیر را مقدار دهی کنید :

MYSQL_DUMP_PATH & MYSQL_PATH - اگر از دیتابیس مای اس کیو ال استفاده می کنید
SQL_CMD_PATH - اگر از دیتابیس اس کیو ال سرور استفاده می کنید
SQLITE_CLI_PATH - اگر از دیتابیس اس کیو لایت استفاده می کنید

BACKUPTIME :

زمان پشتیبان گیری بر اساس موقعیت زمانی اپلیکیشن شما و فرمت آن HH می باشد

MIGRATION :

اگر مقدار این فیلد برابر on باشد فریم ورک ، فایل Migrate شما را در database -> Migration -> migration_of_db1.sql پردازش کرده و در دیتابیس متصل شما قرار می دهد .

توجه داشته باشید که این عملیات توسط خط فرمان انجام خواهد شد ؛ پس باید قبل از آن 4 متغیر مسیر زیر را مقدار دهی کنید :

MYSQL_DUMP_PATH & MYSQL_PATH - اگر از دیتابیس مای اس کیو ال استفاده می کنید
SQL_CMD_PATH - اگر از دیتابیس اس کیو ال سرور استفاده می کنید
SQLITE_CLI_PATH - اگر از دیتابیس اس کیو لایت استفاده می کنید

کار کردن با ORM

ORM رد فریم ورک چهار عملیات پایه CRUD (ساخت ، خواندن ، به روز رسانی ، حذف) و دو عملیات اضافه (شمارش ، میانگین) را فراهم می کند .


در کد باکس رو به رو تمامی پرچم های موتور پایگاه داده توضیح داده شده است :
* پرچم (Flag) ثابت از پیش تعیین شده برای راحت تر کردن فهم فریم ورک برای توسعه دهنده می باشد

پرچم های موتور پایگاه داده
/* DB1 - اشاره به پایگاه داده شماره 1 دارد
DB2 - اشاره به پایگاه داده شماره 2 دارد
DB3 - اشاره به پایگاه داده شماره 3 دارد
CREATE - اشاره به عملیات ساخت دارد
READ - اشاره به عملیات خواندن دارد
READ_CONDITION - اشاره به عملیات خواندن بر اساس شرط دارد
UPDATE - اشاره به عملیات به روز رسانی دارد
UPDATE_CONDITION - اشاره به عملیات به روز رسانی بر اساس شرط دارد
DELETE - اشاره به عملیات حذف دارد
DELETE_CONDITION - اشاره به عملیات حذف بر اساس شرط دارد
COUNT - اشاره به عملیات شمارش دارد
COUNT_CONDITION - اشاره به عملیات شمارش بر اساس شرط دارد
AVG - اشاره به عملیات میانگین دارد
AVG_CONDITION - اشاره به عملیات میانگین بر اساس شرط دارد
FREE - اشاره به کوئری آزاد دارد */



نکته : تمامی عملیات ORM باید در یک کلاس مدل که از مدل پایه ارث برده شده به انجام برسد .

عملیات ساخت :

ORM - ساخت
$this->setFields([
'page' => 'any',
'hits' => 'any',
]);

$this->setParameters([
'page' => 'singup',
'hits' => '0',
]);

$this->query("hits", CREATE, DB1);

توضیح :


در دستور اول با فراخوانی متد 'setFields' از کلاس مدل با یک پارامتر (از نوع آرایه) که حاوی نام فیلد در Key و شروط ارزیابی اعتبار مدل در Value بود ، فیلد هامون رو ست کردیم .
توجه داشته باشید که 'any' در شروط ارزیابی اعتبار مدل به معنی این هست که نیازی به ارزیابی اعتبار نیست .


در دستور دوم با فراخوانی متد 'setParameters' از کلاس مدل با یک پارامتر (از نوع آرایه) که حاوی نام فیلد در Key و مقدار فیلد در Value است مقادیرمون رو در فیلد ها بایند کردیم .


در دستور سوم و نهایی با فراخوانی متد query از کلاس مدل با سه پارامتر (به ترتیب از راست به چپ : نام جدول ، پرچم عملیات ، نام پایگاه داده) کوئری رو اجرا کردیم .

عملیات خواندن :

ORM - خواندن
$this->setFields([
'page' => 'any',
'hits' => 'any',
]);

$this->query("hits", READ, DB1, "id DESC", $offset, $limit);

توضیح :


به وسیله متد 'setFields' دو فیلد Page و Hits رو در نظر گرفته ایم تا از اون ها داده دریافت کنیم .


و در نهایت با فراخونی متد query با شش پارامتر (به ترتیب از راست به چپ : نام جدول ، پرچم موتور پایگاه داده ، پرچم دیتابیس مورد نظر ، ترتیب ، آفست شروع و محدودیت دریافت داده) کوئری را اجرا کردیم .

ترتیب صعودی : FieldName ASC
ترتیب نزولی : FieldName DESC
* سه پارمتر آخر اختیاری می باشد

عملیات خواندن بر اساس شرط :

در مثال رو به رو به کمک الگوی طراحی بر اساس شرط داده ها را دریافت و خوانده :

ORM - خواندن یر اساس شرط
$this->setFields(['id' => 'any', 'page' => 'any', 'hits' => 'any', ])->
setConditionFields(['page' => 'any|equal|operator:and', 'hits' => 'any|equal'])->
setConditionParams(['page' => 'singup', 'hits' => '60'])->
query("hits", READ_CONDITION, DB1, "id DESC");

توضیح :


به وسیله متود 'setFields' سه فیلد را درخواست کرده ایم .


سپس با متود setConditionFields شروط و خواص فیلد های شروط را مشخص نموده ایم .
توکن اول ، 'any' به این معنی می باشد که نیازی به ارزیابی اعتبار نیست . توکن دوم 'equal' ، عملگر شرطی است (که میتواند یکی از مقادیر رو به رو باشد : (greater, smaller, equal, like, not_like)
توکن سوم عملگر رابطه ای است (که می تواند یکی از مقادیر رو به رو باشد :(and, or)

* نیازی به عملگر رابطه ای برای شرط آخر نیست


نکته : تمامی خواص ارزیابی اعتبار مدل در آخر این بخش کامل توضیح داده خواهد شد .

بعد از ست کردن فیلد های شروط ، نوبت به بایند کردن پارمتر های شروط با متد setConditionParams می باشد .

و در نهایت با فراخوانی متد query و با ترتیب نزولی بر اساس فیلد 'id' کوئری خودمون رو اجرا می کنیم .


بعد از اجرای کوئری با دستور زیر به راحتی میتوانید نتیجه کوئری رو دریافت کنید :

ORM - خواندن بر اساس شرط (دریافت اطلاعات)
$result = $this->getContent(DB1);

عملیات به روز رسانی

به مثال رو به رو توجه کنید :

ORM - به روز رسانی
$this->setFields(['email' => 'any'])->
setParameters(['email' => 'mahbod@gmail.com'])->
query("users", UPDATE, DB1);

توضیح :


در مثال بالا فیلد 'email' تمامی رکورد ها را با مقدار 'mahbod@gmail.com' به روز رسانی کردیم .

عملیات به روز رسانی بر اساس شرط

به مثال رو به رو توجه کنید :

ORM - به روز رسانی بر اساس شرط
$this->setFields(['email' => 'any'])->
setParameters(['email' => 'mahbod@gmail.com'])->
setConditionFields(['username' => 'required|method:en_digit|limit:5-25|sanitize_space:yes|equal',])->
setConditionParams(['username' => 'mahbod'])->
query("users", UPDATE_CONDITION, DB1);

توضیح :


در مثال بالا فیلد 'email' رکوردی که فیلد 'username' آن برابر با 'mahbod' است را با مقدار 'mahbod@gmail.com' به روز رسانی کردیم .

نکته ارزیابی اعتبار مدل : در این مثال فیلد 'username' رو با متود انگلیسی و اعداد ، بین 5 تا 25 حرف ، زدودن space اضافه ارزیابی کردیم که در پایان این بخش به طور کامل توضیح داده خواهد شد .

عملیات حذف :

در مثال رو به رو تمامی رکورد های موجود در جدول را حذف می کنیم :

ORM - حذف
$this->query('hits', DELETE, DB1);

عملیات حذف بر اساس شرط

به مثال رو به رو توجه کنید :

ORM - حذف بر اساس شرط
$this->setFields(['hits'=> 'any']);
$this->setConditionFields(['hits' => 'any|equal']);
$this->setConditionParams(['hits'=>'mahi']);

$this->query('hits', DELETE, DB1);

عملیات شمارش

ORM - شمارش
$this->query('hits', COUNT, DB1);

$result = $this->getContent(DB1);

عملیات شمارش بر اساس شرط :

ORM - شمارش بر اساس شرط
$this->setConditionFields(['id' => 'any|equal']);
$this->setConditionParams(['id' => '12']);
$this->query('test', COUNT_CONDITION, DB2);
$result = $this->getContent(DB2);

عملیات میانگین :

ORM - میانگین
$this->setFields(['rate' => 'any', ]);
$this->query('post_votes', AVG, DB1);

عملیات میانگین بر اساس شرط :

ORM - میانگین
$this->setFields(['rate' => 'any', ]);
$this->setConditionFields(['post_id' => 'any|equal|operator:and',]);
$this->setConditionParams(['post_id' => $id['id'],]);

$this->query('post_votes', AVG_CONDITION, DB1);

کوئری آزاد :

ORM - کوئری آزاد
$this->freeQuery("UPDATE `users` SET `email` = 'mzero651@gmail.com' WHERE `username` = 'mahbod'", READ_CONDITION, DB1);

$result = $this->getContent(DB1);

توضیح :


در مثال بالا با متد خواندن بر اساس شرط یک کوئری آزاد رو با پایگاه داده شماره یک اجرا کردیم و نتیجه اون رو در متغیر result ریختیم .

ارزیابی اعتبار مدل :

در این تاپیک به توضیح ارزیالی اعتبار مدل و خواص آن می پردازیم .

ORM - ارزیابی اعتبار مدل
$this->setFields(['username' => 'required|method:en_digit|limit:5-25|sanitize_space:yes']);

خاصیت اول به منظور نوع فیلد است :

سه نوع فیلد داریم :

required (نمی تواند خالی باشد)
optional (اختیاری)
any (بدون ارزیابی اعتبار)

صفحه بندی

عملیات محاسبه تعداد صفحات به جهت صفحه بندی راحت تر درست بعد از اجرای کوئری خواندن یا خواندن بر اساس شرط شما که دارای آفست و محدودیت باشد ، انجام می شود .

برای مثال :

صفحه بندی
$this->setFields([
'page' => 'any',
'hits' => 'any',
]);

$this->query("hits", READ, DB1, "id DESC", 0, 10);

$pages = $this->getPagesCount(DB1);

ایمیل سرور

رد فریم ورک سیستمی آسان جهت ارسال ایمیل از طریق سرور SMTP با استفاده از کتابخانه PEAR فراهم میاورد .

برای استفاده از این سیستم ابتدا شما باید اطلاعات اولیه سرور رو در فایل Environment.json پیکربندی کنید

بعد از این که عمل پیکربندی رو کامل انجام دادید ، به کمک سینتکس دستوری زیر می توانید به راحتی ایمیل ارسال کنید :

ایمیل
use Red\MailService\Mail;

$subject = "test";
$target = "mzero651@gmail.com";
$body = "<body><h1>test</h1></body>";
$flag = Mail::HTML_Header;

Mail::send($subject, $target, $body, $flag);

رمز گذاری

رد فریم ورک از سیستم رمز گذاری منحصر به خودش که با کلید امنیتی شما Salt شده بهره می برد.

کلید امنیتی باید در پیکربندی متغیر های محیطی تعیین شود .

قابلیت رمز گذاری و رمز نگاری رشته و آرایه (تا حداکثر 3 بعد) برای شما فراهم گردیده است .


فضای نام کلاس رمز گذاری :

Red\Encryption\Crypter

رمز گذاری / نگاری
// رمز گذاری :
$encrypted_data = Crypter::encrypt($data);

// رمز نگاری :
$decrypted_data = Crypter::decrypt($data);

سیستم رمز گذاری مند POST

یکی از امکانات قابل توجه رد فریم ورک ، سیستم رمز گذاری متد POST هست که برای کسب و کار های کوچکی که توانایی خرید SSL ندارند شدیدا کارآمد هست .
این سیستم تمامی متغیر هایی که از طریق متد POST ارسال میشوند ، ابتدا رمز گذاری ، سپس ارسال و در نهایت در سمت بک اند رمز نگاری می کند .

برای استفاده از این سیستم ابتدا باید گزینه اون رو در متغیر های محیطی فعال کنید .

برای اضافه شدن وابستگی ها به صفحه از متد رو به رو در بخش Head ویو خودتون استفاده کنید :

اضافه کردن وابستگی ها به صفحه
{{ encryption() | raw }}

فرم باید دارای ID و متد آن POST باشد ؛ سپس کار به تنظیم رخداد می کشد :

تنظیم رخداد با جاوا اسکریپت خالص
<form id="form1" action="{{ self() }}" method="post" onsubmit="return RedCryption.encrypt('form1');">

تنظیم رخداد با Red JS
<form id="form1" action="{{ self() }}" method="post">

ارزیابی اعتبار کاربر

ارزیابی اعتبار کاربر یک میان راهی از پیش تعیین شده در رد فریم ورک است .
برای شروع کار با این میان راهی باید فرایند ورود کاربر به اپلیکیشن را شخصی سازی کنید ؛

بعد از ورود یک کاربر به سیستم ، سطح دسترسی اون باید در یک متعیر نشست ذخیره شود تا سیستم بر اساس اون سطح دسترسی محتوای قابل دسترس کاربر رو تعیین کنه .

بعد از گذشت از مرحله قبل حالا نوبت به آدرس دهی متغیر نشست در میان راهی ارزیابی اعتبار میرسه :

آدرس دهی متغیر نشست
self::setAuthenticationVariable('user_logged_in', 'access_level');

در مثال رو به رو نحوه استفاده از میان راهی ارزیابی اعتبار در مسیر یاب رو خواهیم دید :

نحوه استفاده از میان راهی ارزیابی اعتبار کاربر
Router::register("panel/{param1}{?param2}", "GET", "PanelController@panel", ["Authentication:admin,master"]);

درخواست AJAX

درخواست AJAX یک میان راهی از پیش تعیین شده است که شما را قادر می سازد درخواست های HTTP عادی را از درخواست های AJAX تشخیص بدهید .

تمامی درخواست ها با سر برگ (header) ، 'AJAX-Request' از این میان راهی گذر خواهند کرد .

نحوه عملکرد میان راهی درخواست AJAX
Router::register("last-posts-api", "GET", "postController@lastPosts", ["AJAXRequest"]);

شما می توانید برای ارسال سر برگ AJAX-Request از جاوا اسکریپت خالص یا از متد AJAX در Red JS استفاده کنید :

درخواست AJAX به کمک Red JS
Red.ready(function () {
    Red.ajax({
            type: 'GET',
            url: 'http://localhost/,
            contentType: 'text/html',
            success: function () {
                Red.select('#content').innerHTML(this.response)
            },
    });

});

جلوگیری از حملات CSRF

محافظ CSRF یک میان راهی از پیش تعیین شده در رد فریم ورک است .

CSRF (Cross-Site Request Forgery) یک ضعف امنیتی شناخته شده است که به نفوذگر این امکان را می دهد تا درخواستی به صورت نا خواسته بر روی سیستم مشتری اجرا شود .
فرض کنید اپلیکیشن شما مسیر GET ای داره که قراره پسورد کاربر رو از این طریق عوض کنه (توجه داشته باشید که هرگز همچین مسیری پیشنهاد شده نیست ؛ فقط برای مثال) :


http://app.com/user-area/reset-password/?new-password=newpass1232

در صورتی که کاربر به اپلیکیشن وارد شده باشد و نشست فعال داشته باشید و نفوذگر درخواستی از سوی کامپیوتر کاربر ارسال کند (توسط مشاهده کردن یک وبسایت نا امن ، می تواند چند خط کد جاوا اسکریپت باشد) گذر واژه کاربر به راحتی عوض خواهد شد .

برای جلوگیری از این حملات بهترین راه استفاده از توکن CSRF در فرم ها است .

به جهت تولید توکن از متد CSRFToken در ویو خود استفاده کنید ؛ حتما توکن رو در یک hidden input قرار بدهید تا همراه فرم ارسال بشه و از سمت بک اند به آن دسترسی داشته باشید .

نحوه کارکرد توکن CSRF
<input type="hidden" name="CSRF_Token" value="{{ CSRFToken() }}">

نحوه کارکرد میان راهی توکن CSRF :

میان راهی محافط CSRF
Router::register("user-area/reset-password/{new-password}", "GET", "UserController@resetPass", ["Authentication:user", "CSRFToken"]);
