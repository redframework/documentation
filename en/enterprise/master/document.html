Directory Structure

APP :

App is Base Directory Of Application; Contains All of your Controllers, Models, Middlewares and APIs.

CONFIG :

Application Config, Session Config, Router Service Config and Path Variables will be Stored at Config Folder.

DATABASE :

Database Backups, Migration Files, Schema and SQLite Embedded Database file are In Database Folder.

PUBLIC :

Public is an Accessible Directory which you Can access Outside of Server
All Front-End Resources like JS, CSS and etc should be Stored at Public Directory

RESOURCES :

Application Resources like Language JSON Files will be Stored at Resources Directory

ROUTES :

Application Registered Routes

STORAGE :

Application / Framework Cache like Sessions, Template Engine Pre-Compiled and Analytics Files (For Red Analytics) Will be Stored There.

VENDOR :

Composer Directory; Containing Packages and Autoload PSR.

VIEWS :

Template Engine Views; Containing Partial and Main Views of Your Application.

Configuration

To Get Started with Red Framework Configuration You Need to generate a New Secret Key to Secure your Application.

to Do That You Can Open Red Analytics On Your Project and Click on Project Tab -> Generate New Secret Key

All Session Configurations are Available at Config -> Session.php which Can be Customized easily

All Path Variables are Available at Config -> Path.php
*** Never Ever Change or Remove Path Variables of Core of Framework
Pathes Should Be Defined (unCommented) By Developer :

DEFINE NECESSARY PATH VARIABLES - EXAMPLE
// ROOT_DIRECTORY
define('ROOT_DIRECTORY', '/');

// PUBLIC_PATH
define('PUBLIC_PATH', '/public/');

// MYSQL_PATH - If You Use a MySQL Database in your Project
define('MYSQL_PATH', 'D:\Wamp\bin\mysql\mysql5.7.24\bin\mysql');

// MYSQL_DUMP_PATH - If You Use a MySQL Database in your Project
define('MYSQL_DUMP_PATH', 'D:\Wamp\bin\mysql\mysql5.7.24\bin\mysqldump');

// SQL_CMD_PATH - If You Use a SQL Server Database in your Project
define('SQL_CMD_PATH', '"C:\Program Files\Microsoft SQL Server\Client SDK\ODBC\130\Tools\Binn\SQLCMD.exe"');

// SQLITE_CLI_PATH - If You Use a SQLite Database in your Project
define('SQLITE_CLI_PATH', ROOT_PATH . "database" . DS . "SQLite" . DS . 'sqlite3.exe');

Router Configuration is Available At Config -> Router.php
You Can Set Your Default Route (404 View) By Following Syntax :

SET DEFAULT ROUTE
Router::setDefaultRoute(function () { http_response_code('404'); \Red\View::render("Red.NotFoundError"); });

You Can Set Your Access Error (403 View - Occurs when Route Method / User Role is Not Valid) By Following Syntax :

SET ACCESS ERROR
Router::register('error/403', 'GET,POST,PUT,PATCH,DELETE', function (){ http_response_code('403'); \Red\View::render("Red.AccessError"); }, 'All');

to Set your Access Error on Web Server; Open .htaccess File at Root Direcroty and define Your Domain Name

SET ACCESS ERROR - WEB SERVER PART
// Define your Domain Name Instead of localhost
ErrorDocument 403 http://localhost/error/403

MIDDLEWARES

Middlewares of Another Name of Front-Controller in MVC Pattern

A Middleware is A Class which Will Be launched Before the Controller in Routing Process

Middlewares are in App\Middewares Namespace

to Create a New Middleware You Should extend Abstract Class Middleware and Implement it's abstract method -> run()

run method Can receive unlimited parameters and it's return type is Boolean

if Run Method Returns TRUE ; Application will Proceed
if Run Method Return FALSE ; Application will Stop Processing

NEW MIDDLEWARE
< ? php
/** Red Framework Middleware
* Generated By Red Analytics
*
* Date: 26/08/2019
* Time: 05:25:17
* @author RedCoder
*/

namespace App\Middlewares;

use Red\Base\Middleware;

class Test extends Middleware
{
 public function run(... $parameters)  {
 // TODO: Implement run() method.
 }
}

Working with ORM

Red Framework ORM Provides 4 Base Operations CRUD (Create, Read, Update, Delete) and 2 Extra Operations (COUNT, AVG)


All Database Engine Flags (Const. For Easier Usage) :

DATABASE ENGINE FLAGS
/* DB1 - Which Refers to Database Slot 1
DB2 - Which Refers to Database Slot 2
DB3 - Which Refers to Database Slot 3
CREATE - Which Refers to Create Operation
READ - Which Refers to Read Operation
READ_CONDITION - Which Refers to Read Operation By Condition
UPDATE - Which Refers to Update Operation
UPDATE_CONDITION - Which Refers to Update Operation By Condition
DELETE - Which Refers to Delete Operation
DELETE_CONDITION - Which Refers to Delete Operation By Condition
COUNT - Which Refers to Count Operation
COUNT_CONDITION - Which Refers to Count Operation By Condition
AVG - Which Refers to Average Operation
AVG_CONDITION - Which Refers to Average Operation By Condition
FREE - Which Refers to Free Query */



Note : All ORM Operations Should be Done in A Model

Create Operation :

ORM - CREATE
$this->setFields([
'page' => 'any',
'hits' => 'any',
]);

$this->setParameters([
'page' => 'singup',
'hits' => '0',
]);

$this->query("hits", CREATE, DB1);


Explain :

At The First Command We Called 'SetFields' Method of Model Class With 1 Parameter (@array) Which Contains Key as Field Name and Value as Model Validation Roles ; We Have Set 'any' for Roles Which Means Anything Would Be Accepted and No Validation is Needed.


At The Second Command We Called 'SetParameters' Method of Model Class With 1 Parameter (@array) Which Contains Key as Field Name and Value as Field Value ;


At Third and Final Command We Called Query Method of Model Class With 3 Parameters (Parameters in ltr Sort : Tables, Operation Flag, Database Name Flag)

Read Operation :

ORM - READ
$this->setFields([
'page' => 'any',
'hits' => 'any',
]);

$this->query("hits", READ, DB1, "id DESC", $offset, $limit);


Explain :

We Have Set Page and Hits Fields to Fetch Data from with SetFields Method .


At Final Command We Call Query Method with 6 Parameters (in ltr Sort : tables, CRUD Flag, Database Slot Flag, order, offset, limit).

Order By Descending : FieldName DESC
Order By Ascending : FieldName ASC
Last 3 Parameters are Optional.

Read Operation By Condition :

In Following Example We Fetch Data By Condition in an Easier Way with Design Pattern :

ORM - READ_CONDITION
$this->setFields(['id' => 'any', 'page' => 'any', 'hits' => 'any', ])->
setConditionFields(['page' => 'any|equal|operator:and', 'hits' => 'any|equal'])->
setConditionParams(['page' => 'singup', 'hits' => '60'])->
query("hits", READ_CONDITION, DB1, "id DESC");


Explain :

We Have Requested 3 Fields with setFields Method .


After that We have Set Our Condition Fields and Attributes with setConditionFields
1st token 'Any' Means no Validation is Needed, 2nd token Named equal is Conditional Operator (greater, smaller, equal, like, not_like)
3rd token is Relational Operator (and, or, ...)

There is No Relational Operator is Needed for Last Condition


Note : Attributes and Model Validation Will Be Introduced Totally at the End of this Chapter

after Setting Conditions it's time to set Condition Parameters with setConditionParams method

and finally we Call Query method which execute our query hits table by id descending order.


After Querying you Can Fetch Value easily with Following Syntax :

ORM - READ_CONDITION (FETCH DATA)
$result = $this->getContent(DB1);

Update Operation :

In Following Example We Update Record in Database :

ORM - UPDATE
$this->setFields(['email' => 'any'])->
setParameters(['email' => 'mahbod@gmail.com'])->
query("users", UPDATE, DB1);


Explain :

At Above Example We Updated any Email Field to 'mahbod@gmail.com' .

Update Operation By Condition :

In Following Example We Update Record in Database By Condition and Validation :

ORM - UPDATE_CONDITION
$this->setFields(['email' => 'any'])->
setParameters(['email' => 'mahbod@gmail.com'])->
setConditionFields(['username' => 'required|method:en_digit|limit:5-25|sanitize_space:yes|equal',])->
setConditionParams(['username' => 'mahbod'])->
query("users", UPDATE_CONDITION, DB1);


Explain :

At Above Example We Updated Email Field to 'mahbod@gmail.com' where username is 'mahbod' .

Model Validation Note : we Validate username field with Following Attributes ; english and digits format, between 5-25 letters, sanitize extra space to match case

Model Validation will be Totally introduced at the End of this Chapter

ADD DEPENDENCIES TO YOUR PAGE

<form id="form1" action="{{ self() }}" method="post" onsubmit="return RedCryption.encrypt('form1');">

<form id="form1" action="{{ self() }}" method="post">

Organized Language Service

Red Framework provides Organized Language System to Develop International Applications.

With Language System You Are Available to provide different Text Messages (Errors, Events and etc).
Default Language should be Configured at Environment.json .

Application Texts are Stored At :

resources -> Language -> Your Language Directory (En, Fa and etc) -> Your Language JSON File .

Namespace for Language Class

Red\LanguageService\Language

LANGUAGE
// Set Current Language :
Language::set('en');

// Get Current Language :
Language::getCurrentLanguage();

// Get Language Field :
Language::get($no, $field);

Picture Tools

One of Red Framework Classes is Picture Tools Which Allows you To Upload / Resize Your Images Safely .

Picture Class is Stored at Following Namespace :

Red\PictureTools\Picture

UPLOAD IMAGE
Picture::uploadImage($file, $upload_path, $new_file_name, $target_width, $target_height);

RESIZE IMAGE
Picture::imageResize($image_resource_id, $width, $height, $target_width, $target_height);

Authentication

Authentication is Pre-designed Middleware in Red Framework .
To Start Working with Authentication You Have to Modify your Login Process .

after a User Logged, User Role should be Stored in a Session Variable .

You Have to Refer Session Variable to Authentication Middleware By Following Syntax :

SET AUTHENTICATION VARIABLE
self::setAuthenticationVariable('user_logged_in', 'access_level');

After You Modified Your Login and Authentication Variable You are Ready to Use Authentication Middleware on Your Routes .

AUTHENTICATION MIDDLEWARE USAGE
Router::register("panel/{param1}{?param2}", "GET", "PanelController@panel", ["Authentication:admin,master"]);

AJAX Request

AJAX Request is Pre-designed Middleware in Red Framework which Allow you to Recognize AJAX Requests from Normal HTTP-Requests.

All Requests with AJAX-Request Header would be Accepted on this Middleware .

AJAX-REQUEST MIDDLEWARE USAGE
Router::register("last-posts-api", "GET", "postController@lastPosts", ["AJAXRequest"]);

You Can Send AJAX-Request Header with Native JS or using RedJS :

AJAX-REQUEST WITH RED JS
Red.ready(function () {
    Red.ajax({
            type: 'GET',
            url: 'http://localhost/,
            contentType: 'text/html',
            success: function () {
                Red.select('#content').innerHTML(this.response)
            },
    });

});

CSRF Token

CSRF Protection is Pre-designed Middleware in Red Framework .

CSRF (Cross-Site Request Forgery) is a Known Vulnerability Which Allows Attacker to Execute an Unwanted Request on Client System .
Imagine that you have a Get Route in your application which Set up New Password for User (**i ve never suggest this kind of routes , its just for example **) :

http://app.com/user-area/reset-password/?new-password=newpass1232

if User has Authenticated ID on Your Application and Attacker do a Request to Your Website from User Computer (By Visiting a Site, it could be a Simple Javascript Code) password will be changed to 'newpass1232' easily .

To Prevent CSRF Attacks we use CSRF Token on Our Forms.

Use CSRFToken method in your Form and Place it in a Hidden text Input .

CSRF TOKEN USAGE
<input type="hidden" name="CSRF_Token" value="{{ CSRFToken() }}">

Using CSRF Token in Route as Middleware :

CSRF PROTECTION MIDDLEWARE USAGE
Router::register("user-area/reset-password/{new-password}", "GET", "UserController@resetPass", ["Authentication:user", "CSRFToken"]);
